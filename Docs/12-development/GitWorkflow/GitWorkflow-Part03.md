# Purpose

This part defines pull requests, the review checklist, and the merge policy. The PR is the quality gate where the cheap model's output is verified before it reaches `main`.

# Pull Request Rules

- Every change to `main` MUST be a PR; direct push is forbidden.
- PR title MUST follow Conventional Commits using the primary change type/scope.
- PR description MUST state: what changed, why, how to test, and any AI-assist notes (e.g. "generated by DeepSeek V4 Flash, reviewed for invoke boundary").
- PRs MUST be small. Large PRs (hundreds of files) MUST be split; the cheap model MUST open incremental PRs per feature/step.
- Draft PRs are encouraged while a multi-step feature is in progress, then marked ready when the gate passes.

# Review Checklist (MUST pass)

- Lint, format check, and `tsc` strict pass.
- Unit tests pass; new logic has tests where required by [[TestingRules-Part01]].
- No direct `invoke` in UI; no business logic in components.
- No hardcoded tokens/colors; uses design system.
- No circular dependencies; feature isolation respected.
- Rust changes remain within the thin-bridge mandate.
- No secrets committed.

# Merge Policy

- Squash merge is preferred so `main` history stays linear and readable.
- The merge MUST be blocked until the CI gate is green and at least one human (or designated reviewer) approves.
- After merge, the feature branch MUST be deleted to avoid stale branches.
