# ADR-017: The LockManager Arbitrates Concurrent Edits

## Status

Accepted.

## Context

Many workers work in parallel, and even with Artifacts and a single MergeManager (ADR-015, ADR-016), two workers can still target the same file or the same function at overlapping times. If nothing arbitrates this, merges collide, work is wasted redoing conflicting changes, and the graph's parallelism becomes a liability instead of an advantage. Locking the entire project would serialize everything and destroy the parallelism that is the whole point of fanning out workers. Locking whole files is coarse: two workers editing two different functions in one file should be able to proceed at once.

Eulinx needs a concurrency-control authority that prevents unsafe overlap while preserving as much parallelism as possible, and that also covers resources beyond files — artifacts, terminals, and runtime objects.

## Decision

A dedicated **LockManager** arbitrates concurrent modification. It grants and denies locks over files and, where feasible, finer-grained units such as symbols (a specific function), so two workers editing different symbols in the same file can proceed simultaneously while two targeting the same symbol are serialized. It also guards other shared resources — artifacts, terminals, runtime objects — from unsafe concurrent modification. Workers and the MergeManager consult the LockManager before touching a shared resource; a denied request waits rather than proceeding. The LockManager is deterministic runtime infrastructure (ADR-018), not an AI.

## Consequences

Positive:

- Parallel workers stay safe without serializing the whole project, preserving the throughput that multi-worker fan-out is meant to deliver.
- Symbol-level locking allows genuine concurrent editing within a single file, which is a meaningful scalability win.
- Centralizing lock decisions makes deadlock avoidance, timeouts, and lock release on worker death solvable in one place.

Negative:

- Symbol-level locking requires understanding code structure enough to identify symbols, which is more complex than file locking and language-dependent; the system may fall back to file locks where symbol resolution is not available.
- Lock lifecycle must be airtight: a worker that crashes holding a lock must have it reclaimed, or the resource stalls forever.
- Waiting on locks can slow a run if many workers contend for the same hot files; contention must be observable and, ideally, reduced by good work partitioning upstream.

## Alternatives Considered

- **No locking; rely on merge conflict resolution alone.** Rejected because it wastes work (both workers finish, then one is discarded) and pushes all coordination onto the merge step, which is more expensive and error-prone than preventing the overlap.
- **Whole-project lock.** Rejected because it serializes everything and eliminates the parallelism that justifies the worker model.
- **Whole-file locking only.** Acceptable as a fallback but rejected as the ceiling, because it needlessly blocks non-overlapping edits within a file; symbol-level is the target where feasible.
