# ADR-004: SQLite as the Primary Local Store

## Status

Accepted.

## Context

Eulinx is local-first: the app, its graphs, history, memory, artifacts, settings, and run state all live on the user's machine, and the app must work fully offline with no required server. It stores a great deal of structured, relational data — workspaces, projects, workers, tasks, workflows, nodes, edges, executions, artifacts, messages, sessions, permissions, plugin grants — and it must persist run state after every state change so an app restart *resumes* a run rather than restarting it (a requirement the WorkflowEngine depends on).

The store must be embeddable in a desktop app with zero setup for the user, transactional and durable, queryable with real relational joins, and reachable from Rust. It also must not require a background server process, because that would contradict the local-first, low-overhead posture from ADR-001.

## Decision

Eulinx uses **SQLite** as its primary local data store, accessed from Rust via **SQLx**. SQLite holds all structured relational data and the authoritative run state of workflows. It is the source of truth that the in-memory engine mirrors and the UI subscribes to. Vector data is *not* stored here (see ADR-005), and full-text search indexing is handled separately (see ADR-021); SQLite remains the relational spine everything else references by id.

## Consequences

Positive:

- Zero-setup, serverless, single-file storage that ships inside the app and works offline, matching local-first and low-overhead goals.
- Real transactions and joins make the relational object model (workspace → project → worker → task → artifact) natural to persist and query.
- Persisting run state after every node state change gives crash recovery and is a precondition for deterministic replay (ADR-020).
- SQLx from Rust keeps database access in the thin native layer with compile-time-checked queries, reducing a class of errors the cheap model would otherwise introduce.

Negative:

- SQLite is single-writer; heavy concurrent writes from many workers must be serialized through the runtime rather than hammering the database from everywhere, which shapes how services persist.
- It is not a vector database and not a full-text engine, so two adjunct stores (LanceDB, Tantivy) are required, adding operational surface.
- Schema migrations must be handled with discipline as the object model evolves, since the store is on the user's machine and must upgrade in place without data loss.

## Alternatives Considered

- **A server database (PostgreSQL/MySQL).** Powerful and concurrent. Rejected because it requires a running server process, contradicting local-first, offline, and low-overhead goals, and burdening the user with setup.
- **Flat files / JSON on disk.** Simple and human-readable. Rejected because the object model is deeply relational and needs transactional integrity and joins; hand-rolled file storage would reinvent a database badly and endanger run-state consistency.
- **An embedded document store.** Considered, but the data is strongly relational and benefits from SQL joins and constraints; a document store would push relational logic into application code, which is worse for a cheap model to maintain.
