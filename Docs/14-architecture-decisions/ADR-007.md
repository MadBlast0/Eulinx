# ADR-007: Plugins Are Sandboxed and Untrusted by Default

## Status

Accepted.

## Context

Eulinx is extensible: plugins add node types, tools, providers, and panels, and a marketplace distributes them. Every other section of the vault describes code Eulinx's authors wrote and reviewed. The plugin system is the one place where code the authors did *not* write, did *not* review, and cannot vouch for runs on the user's machine — next to that user's source code, credentials, and AI provider keys. The realistic threat is not a buggy plugin; it is a plugin authored specifically to exfiltrate a repository and API keys, published under a plausible name.

The easiest possible implementation — dynamically importing plugin code into the Eulinx host process — is also the most catastrophic, because an in-process plugin inherits the host's filesystem access, the SQLite handle, the network, the provider keys, and the process itself. Convenience here is directly opposed to safety.

## Decision

**Plugins are untrusted third-party code and run sandboxed by default.** Plugin code never executes in the Eulinx host process. It runs in a separate OS process with no ambient authority: no filesystem handle, no database handle, no network socket, no inherited environment, no process-spawning ability. A plugin declares every capability it wants in its manifest before installation; the user grants those explicitly, having seen them; and enforcement happens at an RPC broker on *every* call against the stored grant, not once at install. The system fails closed: any ambiguity in manifest, permission, signature, or RPC resolves to denial. Every call into plugin code has a hard timeout and a fail-closed default, so a plugin can neither stall nor crash the core, and no plugin can observe or affect another.

## Consequences

Positive:

- A malicious plugin's blast radius is bounded to what the user explicitly granted; it cannot silently read the repo or steal keys.
- A plugin that hangs, throws, leaks, or dies takes only itself down; the runtime records it and continues.
- Cross-plugin isolation removes a supply-chain attack primitive (one plugin discovering or messing with another).
- The security posture is uniform and legible: the RPC broker is the one doorway, and the PermissionManager is the one authority.

Negative:

- A sandboxed, RPC-mediated plugin is materially more work to build than an in-process one, and the SDK must be a proxy layer of RPC stubs rather than real handles.
- Every plugin call pays serialization and IPC cost, and blocking hooks add latency bounded by their timeouts.
- Some capabilities plugins might want are simply not offered because they cannot be offered safely; this constrains what plugins can do.

## Alternatives Considered

- **In-process plugins (dynamic import/require).** Vastly easier and faster. Rejected absolutely: it hands untrusted code the host's full authority — the exact outcome the whole section exists to prevent. There is no partial version of this rule.
- **Trust plugins from a curated marketplace only.** Rejected because curation is not enforcement; a review process cannot be relied on to catch a well-disguised malicious plugin, and users can side-load.
- **Runtime permission-escalation prompts.** Seemingly user-friendly. Rejected because they convert install-time informed consent into fatigue-driven click-through exactly when a malicious plugin manufactures urgency.
