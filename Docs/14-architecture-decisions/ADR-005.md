# ADR-005: LanceDB for Vector Memory

## Status

SUPERSEDED (2026-07-20) — see Amendment below.

## Context

Eulinx's memory system needs semantic retrieval: given a task, find the most relevant prior artifacts, notes, knowledge-base chunks, and summaries by *meaning*, not by keyword. This powers context injection (feeding a worker the smallest useful context) and the knowledge base (retrieving from uploaded docs and repositories). That requires storing embeddings and performing fast approximate-nearest-neighbor search over them, locally, offline, and at a scale that can grow as a workspace accumulates history.

SQLite (ADR-004) is the relational spine but is not a vector engine; bolting vector search onto it would be slow and awkward. The vector store must be embeddable (no server), reachable from the local app, performant for ANN queries, and able to sit alongside SQLite without becoming a second source of truth for relational facts.

## Decision

Eulinx uses **LanceDB** as its embedded vector store for all vector memory: embeddings for artifacts, memory summaries, and knowledge-base chunks, and the ANN search that context injection and semantic retrieval depend on. LanceDB stores vectors and the minimal metadata needed to link a vector back to its authoritative record in SQLite by id. Relational truth stays in SQLite; LanceDB is an index into it, not a parallel database of record.

## Consequences

Positive:

- Fast local semantic search enables the memory system to inject *relevant* context instead of dumping raw transcripts, directly serving the cost-control goal.
- Embedded and offline, matching local-first and low-overhead posture; no server to run.
- Clear separation of concerns: SQLite owns relational facts, LanceDB owns vectors, and vectors reference SQLite ids rather than duplicating content.

Negative:

- A third store to operate, back up, and keep consistent with SQLite; deletions and redactions must propagate to LanceDB so a "forgotten" record does not linger as a searchable embedding.
- Embedding generation has cost and latency and depends on a provider or local model; the pipeline must decide when to embed (on artifact creation, on summary) and handle failures.
- Consistency between the relational record and its embedding must be maintained; a stale or orphaned vector must not surface content that no longer exists.

## Alternatives Considered

- **Vector search inside SQLite (an extension).** Keeps one store. Rejected because dedicated ANN performance and index quality are materially better in a purpose-built vector store, and Eulinx's semantic retrieval is on the hot path of every context injection.
- **A hosted/cloud vector database.** Powerful and managed. Rejected because it breaks local-first and offline guarantees and would send memory content off the user's machine, contradicting the privacy posture.
- **Keeping embeddings in flat files with a hand-rolled index.** Rejected as reinventing a vector database with worse performance and correctness, which a cheap model could not maintain.

## Amendment (2026-07-20)

The original decision to use **LanceDB** for vector storage has been **deferred**. The actual implementation uses an in-memory TypeScript `VectorMemoryStore` (`src/memory/memory-vector.ts`) backed by `Map` with an `EmbeddingService` (`src/memory/embedding-service.ts`) that produces deterministic hash-based dense vectors (256-dim, FNV-1a hashing, bag-of-words + n-grams) as a local fallback, with optional OpenAI-compatible provider backend. The `MemoryManager` in `src/memory/memory-manager.ts` orchestrates STM, LTM, episodic, semantic, working, and vector memory stores, plus a `KnowledgeBase` for ingestion. Hybrid search combines cosine similarity (over real computed vectors) with keyword scoring and scope / sensitivity filtering.

LanceDB is deferred because the current in-memory approach is sufficient at the local-first scale Eulinx targets today: embeddings are computed synchronously on ingestion, stored in-memory alongside records, and the hash-based local backend produces meaningful cosine similarity without requiring a separate database process or native dependency. A dedicated vector database will be reconsidered when:
- The vector index exceeds ~100k records
- Persistent disk-backed ANN indexes are required for performance
- Distributed execution demands a shared vector store
