# ADR-009: Memory Is Scoped, Summarized, and Redacted

## Status

Accepted.

## Context

Eulinx runs many workers that must "know what each other is doing" without every worker receiving every prior conversation. The naive approach — a single global chat history injected into every context window — is fatal on three fronts. It explodes token cost and context length, which is ruinous for an app whose whole economy depends on cheap models. It leaks: one project's data, or a secret printed to a terminal, would flow into unrelated workers. And it confuses: a worker doing authentication does not benefit from the full transcript of frontend work, it benefits from the relevant summary and the specific upstream artifact.

Eulinx also spans multiple isolated workspaces (ADR-024), and workers touch source code, credentials, and command output that may contain secrets. Memory therefore cannot be a flat, global, raw store.

## Decision

Eulinx's memory is **scoped, summarized, and redacted**. Memory is partitioned by scope (workspace, project, session, execution, orchestrator, task, worker) and respects workspace boundaries as a hard wall. When context is assembled for a worker, it receives the *smallest useful* package: the task, relevant channel summaries, and the specific upstream artifacts — not the entire history. Long histories are summarized before injection rather than copied raw, and references to Artifacts are preferred over inlined content. Memory does not expose secrets by default; sensitive reads are auditable; and memory is deletable and forgettable per retention policy, with deletion propagating to derived stores (including vector embeddings, per ADR-005).

## Consequences

Positive:

- Token and context costs stay bounded even with many workers, directly protecting the cheap-model economics.
- One project's or worker's data cannot silently bleed into another, upholding isolation and privacy.
- Workers get signal, not noise, which improves output quality per token.
- Redaction and auditability make it defensible to run autonomous workers over sensitive code.

Negative:

- Assembling the "smallest useful" context is real engineering: summarization, relevance retrieval, and scope resolution all cost work and can occasionally omit something useful.
- Summarization itself consumes model calls and can lose detail; the system must balance fidelity against cost.
- Redaction must be applied consistently everywhere memory is read, and deletion must reach every derived store, or the guarantees are only partial.

## Alternatives Considered

- **One global chat history injected everywhere.** Simplest. Rejected because it explodes cost and context, leaks across scopes, and drowns workers in irrelevant noise.
- **Unrestricted cross-worker shared state.** Rejected because it breaks isolation and makes reasoning about what any worker knew impossible, which also poisons replay and auditing.
- **No summarization, only truncation.** Rejected because blind truncation drops arbitrary information; summarization preserves the useful facts within a budget.
