# ADR-026: Design-System-First, Token-Driven UI

## Status

Accepted.

## Context

Eulinx is a large, panel-heavy application — canvas, terminals, sidebars, history, git, settings, overlays — and it must support light/dark themes with runtime switching. Most of the UI is written by a cheap coding model. If that model hardcodes colors, spacing, radii, and typography inline across hundreds of components, the result is unmaintainable: theming becomes impossible, visual consistency erodes, and a design change means editing everything. Cheap models are especially prone to sprinkling magic values everywhere unless there is a single, obvious place those values come from. The user explicitly asked for global, centralized design foundations set up *before* building features, so UI is easy to manage and change.

The choice is between letting each component style itself ad hoc versus establishing a centralized token system and component wrappers first.

## Decision

Eulinx is **design-system-first and token-driven.** All design values — colors, typography, font families/sizes/weights, line heights, spacing, radii, borders, shadows, opacity, z-index layers, animation durations and curves, breakpoints — are defined once as global tokens. Components consume tokens, never hardcoded values; there are no inline styles and no hardcoded colors, spacing, radii, or typography. Eulinx builds on Tailwind CSS + shadcn/ui as the primitive base and wraps them in its own global components, providers (theme, overlay, toast, etc.), and utilities. Themes (light/dark) are token sets, switchable at runtime. This foundation is established before feature UI is built.

## Consequences

Positive:

- Runtime theming and consistent visuals fall out naturally because every component reads the same tokens.
- The cheap model has one obvious source for design values, which curbs its tendency to hardcode and keeps the UI coherent.
- A design change is a token change, not a sweep across hundreds of files.
- Building on Tailwind + shadcn/ui avoids reinventing every primitive while still giving Eulinx its own controllable wrapper layer.

Negative:

- Up-front investment: tokens, wrappers, and providers must be built before feature UI, which delays visible features slightly at the start (a trade the user explicitly accepted).
- Discipline is required forever; a single hardcoded color in a component is a small crack, and the cheap model must be steered to use tokens via lint rules and the wrapper components.
- A wrapper layer over shadcn/ui adds indirection; wrappers must stay thin and not fork the primitives unnecessarily.

## Alternatives Considered

- **Style components ad hoc as they are built.** Fastest early. Rejected because it makes theming impossible, produces inconsistency, and turns every later design change into a massive edit — exactly what the user wanted to avoid.
- **Use shadcn/ui directly with no Eulinx wrapper layer.** Rejected because Eulinx needs a controllable seam to enforce tokens, swap primitives later, and add app-specific behavior without editing every call site.
- **A heavier all-in-one component framework instead of Tailwind + shadcn/ui.** Rejected because it would constrain customization and is less idiomatic for the cheap model than the Tailwind + shadcn/ui base the rest of the stack assumes.
