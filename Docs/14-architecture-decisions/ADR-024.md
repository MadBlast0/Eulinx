# ADR-024: Workspace Isolation as a Hard Boundary

## Status

Accepted.

## Context

A Eulinx workspace is a project folder the user selects, and Eulinx creates an isolated environment scoped to it. Several workspaces can exist, each independent. The user's central expectation is that work on one project cannot damage or leak into another: a worker building project A must not read project B's files, mutate project B's code, or pull project B's memory into its context. Because Eulinx runs autonomous workers with filesystem access, terminals, and memory, a weak boundary here is not a minor bug — it is data leakage and cross-project corruption. Agents also must, by default, operate within the selected project folder and not roam the user's wider filesystem.

The boundary must hold across every subsystem at once: files, memory, artifacts, history, locks, and permissions.

## Decision

**Workspace isolation is a hard boundary enforced across all subsystems.** Each workspace has its own scoped state; a worker's filesystem access is confined to its workspace's project folder by default; memory is partitioned by workspace and does not cross the boundary (ADR-009); artifacts, history, locks, and permission profiles are workspace-scoped. Path boundaries are enforced by the WorkspaceManager, and any access outside the selected folder requires explicit permission (ADR-019) rather than being ambient. No subsystem may treat data as global that belongs to a workspace.

## Consequences

Positive:

- One project cannot corrupt or read another, meeting the core user expectation and making it safe to run many projects.
- Scoping memory, artifacts, and locks per workspace also bounds cost and complexity — a worker only ever deals with its own project's data.
- The default "stay inside the project folder" rule limits the damage a confused or manipulated worker can do to the user's wider machine.

Negative:

- Enforcement must be present in every subsystem; a single service that forgets the boundary (a memory query that spans workspaces, a path check that is missing) breaks the whole guarantee.
- Legitimate cross-workspace features (shared templates, cross-project history views) must be built as explicit, controlled exceptions rather than falling out of a global store, which is more work.
- Path-boundary enforcement for terminals and tools, especially with symlinks and relative paths, is fiddly and must be robust against escape.

## Alternatives Considered

- **A single global state with workspace tags.** Simpler to build. Rejected because tag-based separation is easy to get wrong and one missing filter leaks or corrupts across projects; isolation must be structural, not a query convention.
- **Soft isolation (convention, no enforcement).** Rejected because autonomous workers with filesystem access make an unenforced boundary meaningless.
- **One workspace at a time only.** Rejected because users legitimately keep multiple projects, and forcing serial single-workspace use is a poor experience; isolation, not exclusivity, is the right answer.
