# ADR-028: Human-Approval Gates on Destructive Actions

## Status

Accepted.

## Context

Eulinx's workers act autonomously, and most of what they do is safe and reversible: producing artifacts, running tests, reading files. But some actions are destructive or irreversible in ways that can seriously harm the user: deleting files, pushing to git, publishing content externally, or otherwise reaching outside the sandbox with real-world effect. Permission gating (ADR-019) decides whether a worker *may* do a class of action at all, but even a worker with permission should not silently perform an irreversible, high-consequence action on the user's behalf without a moment of human confirmation. The whole "you manage AI workers" model depends on the user staying in control at the points that matter.

The tension is autonomy (the value of workers getting on with it) versus safety (the user must not be surprised by an irreversible action).

## Decision

**Destructive and irreversible actions require an explicit human-approval gate.** Actions such as deleting files, pushing to git, and publishing externally pause for user approval before they execute, even when the worker holds the relevant permission. Approval gates are an architectural rule, expressible as a node in a workflow (a Human-approval node) and enforced by the runtime at the point of the destructive action — notably in the MergeManager (ADR-016) for destructive merges. Workers that are blocked on an approval surface the request to the user rather than proceeding or silently failing. This is distinct from and layered on top of permission gating (ADR-019): permission decides *whether the class is allowed*; the approval gate decides *whether this specific irreversible instance proceeds now*.

## Consequences

Positive:

- The user is never surprised by an irreversible, high-consequence action; they confirm the moments that matter, which is essential trust for autonomous workers.
- Approval gates compose into workflows explicitly, so a template can require human sign-off at defined points.
- Layering approval on top of permissions gives two independent safety checks for the most dangerous actions.

Negative:

- Approval prompts interrupt autonomy; too many or poorly-chosen gates cause fatigue, so the set of gated actions must be limited to genuinely destructive/irreversible ones.
- Blocked workers waiting on approval consume a slot and must time out or be cancellable, and the runtime must handle a run that is paused on human input cleanly.
- Deciding what counts as "destructive enough to gate" is a judgment call that must be specified and kept consistent across tools, MCP, and merges.

## Alternatives Considered

- **Rely on permissions alone.** Rejected because a permission grant is a standing decision made once; it does not give the user a look at the specific irreversible action at the moment it would happen, which is exactly what high-consequence actions warrant.
- **A global "auto-approve / YOLO" mode with no gates.** Considered as an opt-in for power users, but rejected as a default; the safe default gates destructive actions, and any auto-approve must be an explicit, informed user choice.
- **Gate everything, destructive or not.** Rejected because it destroys autonomy and causes approval fatigue that trains users to click through without reading — worse than no gate.
