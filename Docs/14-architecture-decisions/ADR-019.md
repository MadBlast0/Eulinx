# ADR-019: Permission-Gated, Fail-Closed Capabilities

## Status

Accepted.

## Context

Eulinx runs autonomous AI workers that can execute commands, read and write files, use git, browse the web, and call tools. An AI that automatically had filesystem, git-push, delete, SSH, Docker, and internet access would be a loaded weapon: a confused or manipulated worker could delete files, push broken code, or exfiltrate data. No capability should be ambient. At the same time, capabilities come from several places — built-in tools, MCP servers, and plugins — and it would be a disaster if each source enforced its own ad-hoc permission logic, or none did.

The user must remain in control of what any worker is allowed to do, and that control must be enforced consistently at the moment of action, not merely declared at setup.

## Decision

Every capability is **permission-gated and fail-closed**, adjudicated by a single **PermissionManager**. A worker holds an explicit permission profile (read files, write files, delete files, git status, git push, run commands, browser, network, and so on); a capability the worker was not granted does not exist for it. Every capability-gated action — whether from a built-in tool, an MCP server, or a plugin — is checked against the granted permission at invocation time, not just at configuration time. Any ambiguity resolves to denial. There is no runtime escalation path that manufactures a permission the user did not grant. This is the same authority the plugin sandbox (ADR-007) relies on, applied uniformly to workers and tools.

## Consequences

Positive:

- A worker's blast radius is exactly what the user granted, so destructive or exfiltrating actions require explicit prior consent.
- One authority (the PermissionManager) means consistent enforcement across built-in tools, MCP, and plugins, with one place to audit.
- Fail-closed defaults mean a bug or ambiguity denies rather than allows, which is the safe direction for a system running untrusted-ish AI actions.

Negative:

- Every action path must route through the permission check; a capability that bypasses it is a security hole, so tool and MCP integration must be disciplined.
- Fine-grained permissions can create friction; the design must offer sensible defaults and grouped grants so users are not overwhelmed, without falling back to "grant everything".
- A legitimately blocked worker may stall waiting on a permission; the system must surface this to the user rather than silently failing or silently escalating.

## Alternatives Considered

- **Grant workers broad capability by default, restrict exceptionally.** Convenient. Rejected because default-allow is the wrong direction for autonomous AI; one mistake becomes catastrophic, and it inverts the safe default.
- **Per-source permission logic (tools, MCP, plugins each enforce their own).** Rejected because it guarantees inconsistency and gaps; a single authority is the only defensible model.
- **Runtime escalation prompts when a worker needs more.** Rejected for the same reason as in the plugin system: it degrades into fatigue-driven click-through at the worst moment.
