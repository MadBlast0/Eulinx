# ADR-015: Workers Write Artifacts, Never the Project Directly

## Status

Accepted.

## Context

Many workers run at once, and several may be working on related code simultaneously. If each worker wrote directly into the user's project files, they would race: one worker overwrites another's edit, a half-finished change from one worker breaks a file another is reading, and there is no point at which an output has been checked before it becomes part of the user's real code. There is also no clean way to verify, review, or roll back a change that was applied the instant a worker produced it.

The user's project is *trusted state* — it is their real code, and it must never be corrupted by unreviewed AI output. Worker output, by contrast, is a *proposal* until something verifies it and something applies it safely.

## Decision

**Workers produce Artifacts; they never write to the trusted project directly.** A worker operates in its own sandbox and emits its result as an Artifact (a plan, code, a patch, JSON, a summary, test results, an image). Artifacts are the unit of exchange between workers and the unit that gets verified. Only after an Artifact is verified does the MergeManager (ADR-016) apply it to the project, arbitrated by the LockManager (ADR-017). No node handler, no worker, and no Builder writes trusted project state; Builders produce Artifacts, Verifiers check them, and the MergeManager alone applies them.

## Consequences

Positive:

- Concurrent workers cannot corrupt each other's work or the project, because they write to sandboxes/Artifacts, not shared files.
- Every change has a checkpoint before it lands: it can be verified, reviewed, and rejected while still only an Artifact.
- Artifacts become a portable, inspectable, replayable currency that memory references, verifiers check, the knowledge base indexes, and the marketplace can share.
- Rollback is natural: an Artifact that was never merged leaves no trace in the project.

Negative:

- An indirection cost: producing a patch and merging it is more steps than editing a file in place, and the Artifact/merge machinery is real infrastructure to build.
- Large artifacts (big diffs, generated files) consume storage and must be managed and pruned.
- Workers must be constrained so they genuinely cannot write outside their sandbox, or the guarantee is only aspirational.

## Alternatives Considered

- **Workers edit project files directly, with locks to prevent races.** Simpler pipeline. Rejected because even with locks there is no pre-application verification checkpoint, no clean rollback, and a crashed worker can leave a half-written file in the real project.
- **Workers propose diffs but apply them themselves after a check.** Rejected because it scatters the "apply to trusted state" authority across many workers instead of one auditable component, undermining ADR-016.
- **Send full conversations between workers instead of artifacts.** Rejected because it explodes tokens and does not give a verifiable, mergeable unit of work (see the memory decision, ADR-009).
