# ADR-025: The UI Never Calls invoke Directly; a Service Layer Sits Between

## Status

Accepted — compliance verified 2026-07-20.

## Context

Tauri exposes the backend to the frontend through `invoke`, which calls a named Rust command. The easy path is to scatter `invoke` calls directly inside React components wherever a component needs backend data or action. That path rots quickly: command names and argument shapes get duplicated across dozens of components, changing a command means hunting every call site, there is nowhere central to add typing, error handling, retries, or logging, and the components become tightly coupled to the exact backend command surface. For a codebase written mostly by a cheap model, this scatter is especially harmful, because the model will happily duplicate an `invoke` call slightly wrong in many places.

There needs to be one narrow, typed seam between the React UI and the Tauri backend.

## Decision

**The UI never calls `invoke` directly.** A dedicated **service layer** (TypeScript) is the only code that calls `invoke`. Components and hooks call typed service functions; those functions call `invoke`, translate arguments and results, handle errors uniformly, and are the single place command names live. TanStack Query (ADR-010) fetches through these services rather than through raw `invoke`. This gives one typed boundary over the IPC surface, consistent with the thin-Rust architecture (ADR-001), where the frontend talks to the backend only through this controlled seam.

## Consequences

Positive:

- Command names and argument shapes live in one place, so a backend change is a service change, not a hunt across components.
- Typing, error handling, retries, and logging attach once at the seam and apply everywhere.
- Components stay decoupled from the exact IPC surface, which keeps them testable and keeps the cheap model from duplicating `invoke` calls inconsistently.

Negative:

- A layer of indirection every backend call must pass through; developers must resist "just calling `invoke` here quickly", which reintroduces the scatter.
- The service layer must be kept in sync with the backend command surface; a drift between them fails at the seam rather than in the component, which is better but still must be maintained.
- More files and structure than the naive approach, a modest cost for the maintainability gained.

## Alternatives Considered

- **Call `invoke` directly in components.** Fastest to write. Rejected because it duplicates command knowledge, couples UI to IPC, and leaves no place for shared typing, error handling, or logging.
- **A generated client from the backend command definitions.** Attractive and could complement this. Not adopted as the decision here because the service layer must exist regardless as the place to add error handling and translation; generation can layer on later without changing the rule.
- **Route everything through a global store's actions.** Rejected because it conflates state management with the IPC boundary; the service layer is about the transport seam, and state tools (ADR-010) build on top of it.

## Compliance Note (2026-07-20)

An audit of key UI files confirms the service layer is consistently used:

- `src/ui/workspace/top-bar.tsx` — uses `windowService` from `@/api/services` (no direct `invoke`)
- `src/ui/workspace/settings-store.tsx` — uses `settingService` from `@/api/services` (no direct `invoke`)
- `src/ui/workspace/fs-client.ts` — uses `fsService` from `@/api/services` (no direct `invoke`)
- `src/ui/workspace/git-client.ts` — uses `gitService` from `@/api/services` (no direct `invoke`)

All four files import from the `@/api/services` barrel and call typed service functions rather than Tauri's `invoke` directly. No violations of the no-direct-invoke rule were found in the audited files. The service modules in `src/api/services/` provide the typed seam over the IPC surface as mandated.
