# ADR-029: Feature-Based Frontend Architecture, No Circular Dependencies

## Status

Accepted.

## Context

The frontend is large — canvas, terminals, workers, workflows, git, files, settings, plugins, marketplace — and almost all of it is written by a cheap coding model working on small, focused tasks. How the code is organized directly determines whether that model can keep contributing reliably. A flat or type-first layout (all components in one folder, all hooks in another, all stores in another) forces the model to touch code all over the tree for a single feature and makes it easy to create tangled, circular dependencies it cannot reason about. The user explicitly asked that every feature own its own components, hooks, types, services, store, and utils, with strict TypeScript, absolute imports, barrel exports, and no circular dependencies.

The choice is between organizing by *technical type* versus organizing by *feature*.

## Decision

Eulinx uses a **feature-based frontend architecture with no circular dependencies.** Each feature owns its own `components`, `hooks`, `types`, `services`, `store`, and `utils`, and shared design-system foundations (tokens, wrapper components, providers, global services) live in shared locations that features consume. Cross-feature imports go through defined public surfaces (barrel exports), and circular dependencies between features are forbidden. Strict TypeScript, absolute imports, and barrel exports are the standing conventions. This composes with the state split (ADR-010), the service seam (ADR-025), and the design system (ADR-026): a feature's store is Zustand/Query, its backend access is through the service layer, and its UI uses shared tokens and wrappers.

## Consequences

Positive:

- A feature is a self-contained unit, so the cheap model can work on one feature with local context and without wandering the whole tree.
- The no-circular-dependency rule keeps the dependency graph acyclic, which is easier for both humans and the model to reason about and refactor.
- Clear ownership boundaries make it obvious where new code goes and reduce accidental coupling across features.

Negative:

- Some duplication or careful shared-module design is needed for genuinely cross-cutting concerns, versus a type-first layout where everything shared sits in one bucket.
- The conventions (feature ownership, barrels, no cycles) must be enforced (lint rules, review), because a cheap model will otherwise create a cross-feature import or a cycle without noticing.
- Deciding whether something is feature-local or shared is a recurring judgment call that must be made consistently.

## Alternatives Considered

- **Type-first layout (global components/, hooks/, stores/).** Familiar in small apps. Rejected because at Eulinx's scale it scatters each feature across the tree, invites circular dependencies, and gives the cheap model too much surface to touch per task.
- **A monorepo of packages per feature.** More isolation. Rejected as over-engineering for a single desktop app; folder-level feature ownership gives most of the benefit without package-management overhead.
- **No enforced structure, organize as you go.** Rejected because an unstructured codebase written mostly by a cheap model degrades fast into tangled dependencies no one can untangle.
