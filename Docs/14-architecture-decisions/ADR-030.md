# ADR-030: Fail-Closed as the Global Default Posture

## Status

Accepted.

## Context

Eulinx is full of situations where the system does not know what to do: an ambiguous plugin manifest, a permission request that does not clearly map to a grant, an unknown node kind, an unsatisfiable port, an invalid edge, an unreachable MCP server, a malformed AI-proposed mutation, a timed-out hook, an exhausted budget. In each of these, the code faces a fork: on uncertainty, allow and continue (fail open), or deny and halt (fail closed). Failing open is more forgiving and keeps things moving, but in a system running autonomous AI workers, untrusted plugins, and irreversible actions, "allow when unsure" is precisely how data gets leaked, projects get corrupted, and money gets spent. Individual sections already state fail-closed rules (plugins, permissions, the engine); this ADR makes it the explicit, global default so no component reinvents the wrong choice.

## Decision

**Fail-closed is Eulinx's global default posture.** When any component cannot resolve a decision with certainty, it denies, halts, or applies a defined safe default rather than allowing or guessing. Concretely: ambiguous plugin manifests, permissions, signatures, or RPC responses resolve to denial (ADR-007); permission ambiguity denies (ADR-019); an unknown node kind, unsatisfiable port, or invalid edge halts the run rather than guessing (the WorkflowEngine); a timed-out or crashed hook applies its fail-closed default; an exhausted budget stops work (ADR-013); an invalid AI-proposed mutation is rejected (ADR-027). Failing open is a deliberate, documented exception, never a default, and never chosen "to keep things moving".

## Consequences

Positive:

- Uncertainty defaults to the safe direction everywhere, which is the only defensible posture for a system with autonomous AI, untrusted plugins, and irreversible actions.
- One global rule means components do not each re-decide open-vs-closed; the answer is known and consistent, which is especially valuable for a cheap model that would otherwise pick inconsistently.
- Security and correctness properties compose: fail-closed at each layer means the whole system degrades safely rather than dangerously.

Negative:

- Fail-closed can be less convenient: legitimate work may be halted or denied on ambiguity that a fail-open system would have let through, so ambiguities must be surfaced to the user or an orchestrator to resolve rather than silently blocked.
- More care is needed to keep the app usable: safe defaults must be chosen thoughtfully so fail-closed does not turn into "constantly stuck", and blocked states must be observable and actionable.
- Any genuine fail-open exception must be explicitly justified and documented, which is extra discipline (and exactly the point).

## Alternatives Considered

- **Fail-open by default, restrict specific dangerous paths.** More forgiving and smoother. Rejected because "allow when unsure" is the root cause of the exact harms — leakage, corruption, runaway spend — that Eulinx's architecture exists to prevent.
- **Decide open-vs-closed case by case with no global rule.** Rejected because without a stated default, components (and the cheap model) will make the choice inconsistently, and the unsafe choice will creep in wherever it is convenient.
- **Fail-closed only in the plugin and permission systems.** Rejected because uncertainty is everywhere (engine, budgets, mutations, MCP), and a partial rule leaves unsafe defaults in the parts not covered.
