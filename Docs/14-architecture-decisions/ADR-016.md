# ADR-016: The MergeManager Is the Only Writer of Trusted State

## Status

Accepted.

## Context

Given that workers produce Artifacts rather than writing the project directly (ADR-015), something must take verified Artifacts and apply them to the user's real project — and it must be exactly one thing. If many components could apply changes, conflict detection, ordering, approval gates, rollback, and history would each be reimplemented inconsistently, and two appliers could still race each other. The moment where an AI-produced change becomes part of the user's trusted code is the single most dangerous moment in the whole system, and it deserves a single, auditable chokepoint.

Multiple workers finishing at once means multiple Artifacts targeting possibly overlapping files, so applying them safely requires conflict detection, lock coordination, and sometimes human approval, all in one place.

## Decision

The **MergeManager is the sole writer of trusted project state.** It takes verified Artifacts (patches, files) and applies them to the project, coordinating with the LockManager (ADR-017), detecting conflicts, ordering merges, invoking approval gates on destructive changes (ADR-028), recording merge history, and supporting rollback. Orchestrators and workers never merge; they hand verified Artifacts to the MergeManager. Every write to the user's real code passes through this one component and is attributable in history.

## Consequences

Positive:

- Exactly one place applies changes, so conflict handling, ordering, approval, history, and rollback are implemented once and correctly.
- The most dangerous operation in the system has a single audit point, which is what makes autonomous multi-worker editing defensible.
- Rollback and merge history are natural because all writes flow through one component that records them.

Negative:

- The MergeManager is a serialization point and a potential bottleneck under many simultaneous merges; it must handle non-conflicting merges efficiently while still serializing conflicting ones.
- It is a critical, high-stakes component; a bug here can corrupt the user's real project, so it warrants the strongest testing and review.
- Orchestrators lose the ability to "just write the file", which is a discipline they must accept.

## Alternatives Considered

- **Let each worker or orchestrator merge its own Artifacts.** Simpler locally. Rejected because it scatters the highest-risk operation across many components, duplicates conflict/approval/rollback logic inconsistently, and reintroduces races.
- **A generic version-control merge (shell out to git) for everything.** Useful for some cases. Rejected as the sole mechanism because Eulinx needs internal, symbol-aware, approval-gated merges with its own history and rollback semantics beyond what a git merge alone provides.
- **No merge step; verified Artifacts auto-apply.** Rejected because it removes conflict detection, ordering, and approval gates, and gives no single point to roll back from.
