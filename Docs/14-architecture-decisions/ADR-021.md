# ADR-021: Tantivy for Full-Text Local Search

## Status

SUPERSEDED (2026-07-20) — see Amendment below.

## Context

Users need to search their workspace by keyword: find a chat, a project, a workflow, an artifact, or a command by the words in it. This is *lexical* search — exact and fuzzy keyword matching, ranking, and highlighting — which is different from the *semantic* similarity search that vectors provide (ADR-005). SQLite (ADR-004) offers only basic text matching and is not a full-text search engine with proper tokenization, ranking, and fast inverted indexes at scale. Keyword search is a distinct capability with distinct performance needs, and it must work locally and offline.

## Decision

Eulinx uses **Tantivy**, a Rust full-text search library, for local lexical search: indexing and querying chats, projects, workflows, artifacts, and other text content by keyword with ranking and highlighting. Tantivy runs in the Rust backend, indexes content that lives authoritatively in SQLite, and returns matches keyed back to SQLite ids. It complements, and does not replace, LanceDB's semantic search: keyword search answers "find the text that contains these words", vector search answers "find the text that means something similar". SQLite remains the source of truth; Tantivy is an index into it.

## Consequences

Positive:

- Fast, ranked, offline keyword search across the workspace, which users expect and which SQLite alone cannot provide well.
- Written in Rust, it fits the backend and keeps the index-building on the native side, close to the data.
- Clean complement to vector search: lexical and semantic search each do what they are good at, and both reference SQLite ids rather than duplicating truth.

Negative:

- A fourth store to keep consistent with SQLite (alongside LanceDB); index updates, deletions, and redactions must propagate so search never surfaces content that no longer exists.
- Index building and maintenance cost CPU and disk; the pipeline must decide when to (re)index and handle large content.
- Two search systems (lexical + semantic) mean the UI/logic must decide when to use which, or combine them, adding some complexity.

## Alternatives Considered

- **SQLite FTS (full-text extension).** One fewer store. Rejected because Tantivy offers stronger tokenization, ranking, and performance for the search experience Eulinx wants, and search is a first-class feature, not an afterthought.
- **Use vector search for everything, including keyword lookups.** Rejected because semantic similarity is poor at exact keyword recall (finding a specific error string, a filename, a command); lexical search is the right tool for that.
- **A hosted search service.** Rejected because it breaks local-first and offline guarantees and would ship content off the machine.

## Amendment (2026-07-20)

The original decision to use **Tantivy** for full-text search has been **deferred**. The actual implementation is a TypeScript in-memory search index (`src/database/search-index.ts`) using an `InMemoryIndex` class with substring-based scoring (term frequency with log-weighting, metadata bonus) and `localStorage` persistence. The index is rebuilt from a callback and supports add/remove/query/clear operations with snippet extraction.

Tantivy is deferred because the current in-memory approach provides adequate keyword search for the local-first desktop use case: documents are indexed in memory on load, queries are resolved by substring matching with ranking, and the index is persisted to `localStorage` for basic durability. A dedicated full-text engine will be reconsidered when:
- The index exceeds ~50k documents
- BM25-style ranking and fuzzy matching are required
- Index persistence beyond browser storage is needed
