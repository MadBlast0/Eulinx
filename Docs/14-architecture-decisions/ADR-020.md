# ADR-020: Deterministic Replay of Every Run

## Status

Accepted.

## Context

Eulinx runs complex, multi-worker, dynamically-growing graphs that a user cannot watch every part of in real time. When something goes wrong — a worker produced bad code, an orchestrator planned poorly, a merge conflict cascaded — the user (and the developer, and the cheap coding model debugging Eulinx itself) needs to understand *what happened* and reproduce it. AI generation is inherently nondeterministic, so a truly bit-for-bit replay of model outputs is impossible without recording them. But the *engine's* decisions — which node ran next, in what order, given the recorded results — can and must be deterministic, so that replaying a run visits the same nodes in the same order.

Replay is also a foundation for debugging, for time-travel inspection, and for trust: a run you can replay step by step is a run you can explain.

## Decision

Eulinx supports **deterministic replay of every run.** The WorkflowEngine (ADR-006) is a deterministic tick loop over persisted state: given the same graph, the same inputs, and the same *recorded* node results, a replay visits the same nodes in the same order. Model outputs, tool results, and other nondeterministic inputs are recorded as they happen (as Artifacts and run-state entries), so replay feeds the engine the recorded results rather than re-invoking models. Every node state change and run state change is persisted, and the engine can reconstruct and re-walk a run exactly. This is why the runtime must be deterministic (ADR-018) and why run state is persisted after every change (ADR-004).

## Consequences

Positive:

- Any run can be reconstructed and re-walked for debugging, explanation, and time-travel inspection, which is invaluable for a system too complex to watch live.
- Replay makes bug reports actionable: a recorded run reproduces the exact engine behavior.
- It reinforces the discipline that engine decisions are deterministic and node handlers only read declared inputs and write declared outputs.

Negative:

- Recording node results, model outputs, and state transitions costs storage; runs must be prunable and the recording must be efficient.
- Replay reproduces the *engine's* path using recorded outputs; it does not re-run models, so it shows what happened, not what a fresh model call would now produce — this distinction must be clear in the UX.
- Determinism forbids node handlers from reading hidden global state or wall-clock nondeterminism; every such shortcut breaks replay and must be prevented in review.

## Alternatives Considered

- **No replay; rely on logs.** Cheaper. Rejected because flat logs cannot reconstruct the engine's decision path or let a user step through a complex, dynamically-grown run.
- **Re-run models to "replay".** Rejected because model nondeterminism means a re-run is a different run, not a replay of the original; recorded results are required for fidelity.
- **Record only final outputs, not intermediate state.** Rejected because debugging a failure needs the intermediate node states and the order in which they changed, not just the end result.
