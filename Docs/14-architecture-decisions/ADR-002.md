# ADR-002: React Flow for the Node-Graph Canvas

## Status

Accepted.

## Context

The center of Eulinx's interface is a living node graph: worker terminals, orchestrators, tool nodes, logic gates, builders, and verifiers appear as nodes; edges carry data and control flow between them; and while work happens, packets animate along edges and nodes change status color. The graph is not static — orchestrators grow it while a run is in progress. This canvas is the product's signature observability surface, so it must be smooth, customizable, and able to render custom node bodies (a minimized chip, a compact status card, or a full xterm.js terminal).

Building a graph canvas from scratch on raw SVG or Canvas would mean reinventing pan, zoom, selection, edge routing, drag-and-drop, viewport culling, and handle/port interaction — a large, bug-prone effort that a cheap coding model would struggle to produce and maintain. The frontend is React 19, so whatever we choose must be idiomatic React and let node contents be ordinary React components.

## Decision

Eulinx uses **React Flow** as the node-graph canvas library. Nodes are custom React components; the terminal, chip, and card views of a worker are all React Flow custom nodes. Edges use React Flow's edge system with custom styling for data edges, control edges, and artifact edges. Crucially, **React Flow state is treated as a view, not as truth**: the authoritative graph lives in SQLite and the WorkflowEngine's in-memory mirror, and the canvas subscribes to it.

## Consequences

Positive:

- Pan, zoom, selection, drag, port/handle interaction, and viewport performance come for free and battle-tested, so the cheap model writes node *content*, not graph *plumbing*.
- Custom nodes are plain React components, which fits the React 19 + TypeScript frontend and the design-system-first approach.
- Animated edges and status-driven node styling — the observability differentiator — are straightforward to express.

Negative:

- React Flow holds its own internal state for layout and interaction, which creates a standing risk that a developer treats it as the source of truth. This is explicitly forbidden (see ADR-008); the authoritative graph is the engine's, not the canvas's.
- Very large graphs (hundreds of live worker nodes) require virtualization and lightweight node bodies (minimized chips) to stay smooth; heavy nodes like full terminals must be rendered only when maximized.
- We take a dependency on a third-party library's release cadence and licensing for the most visible part of the UI.

## Alternatives Considered

- **Hand-built SVG/Canvas graph.** Maximum control. Rejected because it recreates an enormous amount of interaction plumbing that a cheap model cannot reliably build or maintain, delaying every feature that depends on the canvas.
- **A game/2D engine (PixiJS or similar).** Excellent raw rendering performance. Rejected because node contents must be rich, accessible React components (terminals, forms, cards), which a canvas-only engine makes painful, and because it does not integrate with the React component model.
- **A different React graph library.** Considered, but React Flow is the most established, best-documented, and most cheap-model-friendly option for custom-node graphs in the React ecosystem, which matters directly for who is writing the code.
