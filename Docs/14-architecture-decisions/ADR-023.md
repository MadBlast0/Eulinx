# ADR-023: BYOK Secrets Live in the OS Secure Store

## Status

Accepted.

## Context

Eulinx is bring-your-own-key: the user supplies API keys for their model providers, and the only outbound traffic is to those providers. These keys are high-value secrets — leaking them means someone else spends the user's money and accesses the user's provider account. Storing them in plaintext config files, in SQLite rows, or in the frontend's reach would be dangerous: a plugin (untrusted, ADR-007), a leaked backup, or a misbehaving component could read them. The app also runs untrusted-ish AI workers and third-party MCP servers and plugins, so keys must be kept away from all of those. Eulinx itself must not "see" or transmit keys anywhere except to the provider the user configured.

The platform (Tauri, ADR-001) provides access to the operating system's secure credential store, which is the OS-blessed place for secrets.

## Decision

**BYOK secrets are stored in the operating system's secure store**, accessed through the Tauri Rust backend, never in plaintext config, never in SQLite, and never held by the frontend. Keys are read by the Rust layer only when constructing an outbound request to the provider the user configured (through the provider abstraction, ADR-011). Plugins never receive keys (they run sandboxed with no such capability, ADR-007), and no component other than the outbound provider call has access. Eulinx transmits keys only to the user's chosen provider, nowhere else.

## Consequences

Positive:

- Secrets get OS-level protection (Keychain, Credential Manager, or the Linux secret service) rather than living in readable files or database rows.
- Keeping keys in the native layer and out of the frontend, plugins, and SQLite shrinks the attack surface to the smallest possible set of code.
- Reinforces the privacy promise: keys go only to the user's provider, and Eulinx never sees or stores them elsewhere.

Negative:

- OS secure stores differ per platform, so the Rust integration has three code paths to get right — exactly the kind of native code the cheap model handles worst, requiring careful authorship and review.
- Some environments (headless Linux, certain CI setups) have weak or absent secure stores; the design must handle a missing store gracefully without silently falling back to plaintext.
- The frontend never holding keys means every provider call is mediated by the backend, which is more indirection than calling a provider from TypeScript directly, but is the correct trade.

## Alternatives Considered

- **Store keys in a config file or SQLite (optionally encrypted with an app key).** Simpler and uniform across platforms. Rejected because an app-managed key file is far weaker than the OS store and is reachable by backups, other processes, and bugs; the OS store exists precisely for this.
- **Let the frontend hold keys and call providers from TypeScript.** Rejected because it exposes keys to the WebView and anything running there, contradicting the thin-native-secrets posture.
- **Route all provider calls through a Eulinx cloud that holds keys.** Rejected outright because it breaks BYOK, local-first, and privacy: Eulinx would then see the user's keys and traffic.
