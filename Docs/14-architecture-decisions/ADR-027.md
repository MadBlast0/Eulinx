# ADR-027: Dynamic Graphs Are Validated as Untrusted Input

## Status

Accepted.

## Context

One of Eulinx's most distinctive behaviors is that the graph grows while it runs: an orchestrator decomposes a goal into phases, spawns sub-orchestrators, and workers spawn more workers, so the graph is alive and expanding. The mutations that make it grow are proposed by AI, which means they can be malformed, contradictory, or dangerous — an orchestrator might propose a cycle, a node that reads a port that does not exist, an edge into an already-completed node, or a structure that would invalidate work already done. If the engine applied AI-proposed mutations blindly, a single bad plan could corrupt a run's state, break determinism, or invalidate a completed node's result.

AI output is not a trusted API call; it is a proposal that must be checked before it changes the run.

## Decision

**AI-proposed graph mutations are treated as untrusted input and validated structurally before they are applied.** When an orchestrator proposes inserting nodes, expanding a subgraph, or replanning, the WorkflowEngine (ADR-006) validates the mutation: it checks structural validity (no illegal cycles, valid ports, satisfiable edges) and enforces the rule that a mutation must never invalidate an already-completed node's result. A mutation that fails validation is rejected rather than guessed at, and the engine fails closed (ADR-030). Only validated mutations are applied to the authoritative graph.

## Consequences

Positive:

- The graph can grow dynamically — a signature capability — without letting a bad AI plan corrupt run state or break determinism.
- Completed work is protected: no live mutation can retroactively invalidate a node that already finished, so partial results stay trustworthy.
- Rejecting invalid mutations rather than absorbing them keeps the engine's invariants intact and replay honest.

Negative:

- Every mutation path needs a validator, which is real logic and must cover the ways an AI plan can be wrong (cycles, dangling ports, illegal targets).
- A rejected mutation must be handled gracefully — surfaced back to the orchestrator to replan, or to the human — rather than silently dropped, which is more work than blind application.
- Validation adds latency to graph growth, a cost paid for safety and determinism.

## Alternatives Considered

- **Apply AI-proposed mutations directly.** Simplest and fastest. Rejected because a single malformed plan could corrupt the run, break replay, or invalidate completed nodes — the graph's aliveness would become its fragility.
- **Only allow a fixed, static graph, no growth.** Rejected because dynamic, self-expanding graphs are a core differentiator; removing them removes much of what makes Eulinx distinctive.
- **Validate lazily at execution time instead of at mutation time.** Rejected because a bad structure should be caught when proposed, before it pollutes the authoritative graph and confuses everything downstream.
