# ADR-013: Token and Cost Budgets Are First-Class and Enforced

## Status

Accepted.

## Context

Multi-agent orchestration is powerful and expensive: an orchestrator-workers pattern can cost many times the tokens of a single agent, and Eulinx deliberately fans out into many workers and runs refinement loops that iterate. Without a hard limit, a single "build me a SaaS" request could spawn a runaway tree of workers and refinement passes that silently drains the user's provider account. Even for BYOK users paying providers directly, uncontrolled spend is a trust-destroying failure. The refinement loop in particular *must* have a budget as one of its stopping conditions, or it can loop indefinitely.

Cost and token usage are not a reporting afterthought; they are a resource the system consumes exactly like CPU and RAM, and they must be measurable, attributable, and stoppable per run, per worker, and per workspace.

## Decision

**Token and cost budgets are first-class, measured, and enforced.** Every model call's token count and cost are captured through the provider abstraction (ADR-011) and attributed to a worker, task, phase, and workspace. Budgets can be set at those scopes; the runtime consults the budget authority before and during work and stops, denies, or refuses to escalate when a budget is exhausted, failing closed rather than overspending. The refinement loop treats budget exhaustion as a mandatory stopping condition (ADR-014). Per-run cost estimates are shown before expensive runs, and heavy modes warn before executing.

## Consequences

Positive:

- The user is protected from silent runaway spend, which is essential for trust in an app that autonomously fans out workers.
- Budgets give the "quality per dollar" dial teeth: users can cap cost and let cheap models do the bulk, spending only where they allow.
- Attribution enables cost analytics per worker/phase/workspace and honest pre-run estimates.

Negative:

- Every model call path must flow through accounting; a call that bypasses the abstraction escapes the budget, so the provider abstraction must be the only way to reach a model.
- Enforcing budgets mid-run means work can be halted partway, and the system must handle a partially completed run gracefully (report, resume with a higher budget, or roll back cleanly).
- Accurate cost requires up-to-date per-model pricing and token counting, which must be maintained and can only estimate for some providers.

## Alternatives Considered

- **Track usage for display only, do not enforce.** Simplest. Rejected because visibility without enforcement still allows runaway spend and gives the refinement loop no budget-based stopping rule.
- **A single global spend cap only.** Rejected because it cannot express "let this phase spend more, keep that worker cheap", losing the per-scope control that makes budgets useful.
- **Rely on the provider's own limits.** Rejected because provider limits are coarse, out of Eulinx's control, and cannot stop a specific runaway run within the user's overall account.
