# ADR-012: Model Profiles Instead of Raw Model Names

## Status

Accepted.

## Context

Workers, orchestrators, critics, judges, and builders all need to pick a model. If every node and every configuration referenced raw model identifiers, the graph and its templates would be brittle: a template built with one specific model would break for a user who does not have that model, model names change and deprecate, and a casual user has no idea whether they want a particular versioned model string. It also fights the product goal that a template can be shared and "just run" on someone else's setup with their own keys and models.

At the same time, Eulinx's whole economy is about matching the *kind* of model to the *kind* of work — cheap and fast for bulk generation, stronger for a critic, vision-capable for image tasks, offline for privacy. That intent is about roles, not specific vendor SKUs.

## Decision

Eulinx exposes **model profiles** — semantic roles like coding, reasoning, planning, writing, vision, fast, cheap, and offline — instead of forcing users and templates to name raw models. A profile maps, per user and per workspace, to a concrete provider+model via the provider abstraction (ADR-011). Nodes and templates reference profiles; the mapping is resolved at run time from the user's configuration. This lets a shared template say "use the cheap coding profile" and run correctly on any user's own models, and it makes the refinement loop's "cheap generator, stronger critic" pattern (ADR-014) expressible as profiles rather than hard-coded SKUs.

## Consequences

Positive:

- Templates and graphs are portable: they express intent (a role) and resolve to whatever model the user actually has, upholding no-lock-in and "just run" goals.
- The UX is simpler for everyone: pick "cheap" or "reasoning", not a versioned model string only an expert recognizes.
- The cheap-by-default, strong-where-needed strategy becomes a first-class, configurable concept rather than scattered magic strings.

Negative:

- A mapping layer must exist and have sensible defaults, and the user must be able to see and edit which concrete model each profile points to.
- Profiles are an abstraction over real capability differences; a profile promising "vision" must actually resolve to a vision-capable model, so the mapping must be capability-aware, not just a label.
- Power users occasionally want to pin an exact model; the design must allow overriding a profile with a specific model without abandoning profiles everywhere else.

## Alternatives Considered

- **Reference raw model names everywhere.** Direct and precise. Rejected because it makes templates non-portable, breaks on model deprecation, and forces every user to understand vendor SKUs.
- **A single global default model.** Simplest. Rejected because it discards the entire cheap-vs-strong matching that Eulinx's economics and refinement loop depend on.
- **Auto-select the model per task with no user control.** Rejected because users must be able to control cost and privacy explicitly; a hidden auto-router removes the quality/dollar dial that is a selling point.
