# ADR-010: Zustand for UI State, TanStack Query for Server State

## Status

Accepted.

## Context

The frontend has two very different kinds of state, and conflating them is a classic source of bugs. The first is *client/UI state*: which node is selected, which panels are open, theme, layout, ephemeral view preferences, transient terminal-view attachment. This state is owned by the frontend and is synchronous. The second is *server-shaped state*: data that ultimately comes from the backend over Tauri IPC and can be fetched, cached, invalidated, and refetched — lists of workers, tasks, artifacts, history, workspace metadata. This state is asynchronous and has cache-coherence concerns.

Using one tool for both leads either to a global store stuffed with async fetch logic and manual cache invalidation, or to server data scattered through component effects with no shared cache. The frontend is React 19 + TypeScript and, per ADR-008, most *runtime* truth arrives as events; but there is still a clear line between what the UI owns and what it fetches.

## Decision

Eulinx uses **Zustand** for client/UI state and **TanStack Query** for server-shaped/async state. Zustand holds project-scoped UI state — selection, open panels, layout, theme choice, view modes. TanStack Query owns fetching, caching, background refetching, and invalidation of data retrieved from the backend through the service layer (ADR-025). Live runtime truth still flows via the EventBus (ADR-008); TanStack Query and Zustand are for the state that is not a raw live stream, cleanly separating the two rather than forcing one library to do both jobs.

## Consequences

Positive:

- Each library does what it is best at: Zustand is a small, fast, boilerplate-light store for synchronous UI state; TanStack Query handles caching, refetch, and invalidation so the cheap model does not hand-roll cache logic.
- The client/server state boundary is explicit, which prevents the "everything in one global store" tangle and makes each feature's state easy to reason about.
- Both are idiomatic in the React/TypeScript ecosystem the cheap model is strongest in.

Negative:

- Two state tools mean a rule everyone must follow about which state goes where; misplacing server data in Zustand or UI state in Query reintroduces the tangle.
- Coordinating event-driven updates (ADR-008) with Query's cache requires care so an incoming event invalidates or updates the right cached data rather than leaving it stale.
- Zustand's freedom (no enforced structure) can be misused into a dumping ground unless the feature-based structure (ADR-029) is respected.

## Alternatives Considered

- **Redux (with middleware) for everything.** Powerful and structured. Rejected as heavier boilerplate than needed, and it still does not solve server-cache concerns as cleanly as a purpose-built query library; more code for a cheap model to maintain.
- **Only Zustand, hand-rolling fetch/cache.** Rejected because it recreates caching, refetch, and invalidation badly, which is exactly what TanStack Query provides correctly.
- **Only TanStack Query, using it for UI state too.** Rejected because purely synchronous UI state (selection, panels) is not server data and does not belong in a fetching/caching layer.
