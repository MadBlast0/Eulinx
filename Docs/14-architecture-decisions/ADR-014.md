# ADR-014: The Refinement Loop With a Mandatory Stopping Rule

## Status

Accepted.

## Context

Eulinx's signature feature is a refinement control (Low, Medium, High, Ultra) that decides how many critique-then-refine passes an output gets before it is accepted. The premise is honest and specific: iterating on a *cheap* model's rough draft produces measurably better output per dollar, not magic parity with a flagship. Research supports iterate-with-feedback gains and judge-based preference improvements. But an iterative loop has an obvious failure mode: it can run forever, or keep spending on passes that no longer improve the output, or refuse to accept that some tasks a base model simply cannot do no matter how many loops run.

A refinement loop without a firm way to stop is a cost and reliability hazard, and a loop that overclaims its results is a trust hazard.

## Decision

Eulinx implements the refinement loop as **generate → critic → refine → judge**, repeated, with a **mandatory stopping rule**. The loop stops when any of these is true: the mode's maximum iteration count is reached, the judge decides the latest pass did not improve on the previous one, or the token/cost budget (ADR-013) is exhausted. Modes map to iteration depth (Low is a single draft with no refine, up to Ultra with more passes and a stronger critic), and the generator and critic may be different model profiles (a cheap generator with a stronger critic). The UX describes results honestly as "refined, higher-quality", never as "equal to a flagship", and objective verification (ADR-015) remains the authority on correctness, not the judge.

## Consequences

Positive:

- The loop always terminates, bounded by iterations, judged improvement, or budget, so it cannot run away in time or cost.
- Users get a real quality/dollar dial: cheap models for bulk, extra refinement only where it matters.
- Separating generator and critic profiles enables the cost-effective "cheap draft, strong review" pattern.
- Honest framing avoids the trust damage of claiming parity the loop cannot deliver.

Negative:

- A judge is itself a heuristic; deciding "did this improve" is imperfect and can occasionally stop early or continue needlessly.
- More passes cost more, so the mode/budget interaction must be surfaced clearly (Ultra should warn before running).
- The loop can end without a satisfactory result on tasks beyond the base model's ability; the UX must communicate this rather than imply endless looping guarantees success.

## Alternatives Considered

- **A fixed number of passes with no judge.** Simpler. Rejected because it wastes passes when the output already converged and cannot stop early, and it gives no signal about whether refinement helped.
- **Loop until the judge is fully satisfied, no cap.** Rejected because a satisfaction threshold can be unreachable, producing unbounded cost and time — the exact runaway this ADR prevents.
- **Market it as reaching flagship quality.** Rejected as dishonest and a trust risk; the loop improves output per dollar, it does not guarantee parity.
