# ADR-008: The UI Renders Backend Truth via the EventBus

## Status

Accepted.

## Context

Eulinx's interface shows a lot of live state: worker status, terminal output, node progress, run state, budgets, artifacts appearing, merges happening. It is tempting to let the UI hold this state locally and mutate it optimistically — flip a node to "running" in React state when the user clicks run, append to a terminal buffer in the component, track a worker's progress in a store the UI owns. That path leads to two sources of truth that drift: the UI shows one thing while the runtime believes another, and after a crash or restart the UI's picture is simply wrong.

The runtime already must persist authoritative state (ADR-004) and already emits events for every meaningful change through the EventBus. The question is whether the UI is a *co-author* of state or a *subscriber* to it.

## Decision

**The UI is a subscriber, not an author, of runtime state.** Backend services hold the truth (run state in SQLite, live streams from processes), and they emit events on the **EventBus** for every state change — node state changes, run state changes, worker lifecycle, task progress, artifact creation, merge results, budget thresholds. The UI renders that truth and reacts to those events. The React Flow canvas, the terminal views, the history panels, and the status indicators are all views over backend truth. The UI never treats its own local state as authoritative for anything the runtime owns.

## Consequences

Positive:

- One source of truth. What the user sees matches what the runtime believes, including after a crash and restart, because the UI re-derives from persisted state.
- Multiple views stay consistent automatically: a worker's chip, its card, and its terminal all reflect the same emitted events.
- Plugins, logging, replay, and metrics can subscribe to the same event stream the UI does, so observation is uniform.

Negative:

- The UI cannot cheaply "fake" instant feedback for runtime-owned state; it must show a pending/optimistic-but-labeled state and reconcile when the real event arrives, which is more work than a naive local mutation.
- Event volume can be high (terminal streams, many concurrent workers), so events must be batched/throttled and the UI must handle bursts without stalling.
- Discipline is required: any place the UI mutates runtime-owned state directly is a bug, and reviewers must watch for it.

## Alternatives Considered

- **UI-owned optimistic state with backend sync.** Snappy. Rejected because it creates two sources of truth that drift, and it cannot correctly recover the picture after a crash where the UI's local state is gone or stale.
- **Polling the backend for state.** Simple. Rejected because it is laggy for live streams and wasteful under many concurrent workers; event push is both fresher and cheaper here.
- **Rendering directly from React Flow's internal state as truth.** Rejected as a direct violation of ADR-002; the canvas is a view, and its internal state is presentation, not authority.
