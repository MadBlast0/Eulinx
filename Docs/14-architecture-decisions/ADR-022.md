# ADR-022: MCP as the Capability-Extension Surface

## Status

Accepted.

## Context

Agents out of the box cannot browse the live web, read images and video, generate media, publish results, or run deep research. Eulinx must grant these capabilities, but building a bespoke integration for each — a browser integration, a web-search integration, a media integration, a publishing integration — would be an endless treadmill and would tie Eulinx to specific vendors. Meanwhile an open standard, the Model Context Protocol (MCP), already lets tools expose capabilities to AI in a uniform way, and an ecosystem of MCP servers exists that Eulinx could reuse instead of reinventing.

The tension is between building capabilities in-house (full control, high cost, vendor lock-in risk) and adopting an ecosystem standard (immediate breadth, dependence on external servers, and a new trust boundary).

## Decision

Eulinx uses **MCP as its primary capability-extension surface.** A Settings section lets users add, remove, enable, and disable MCP servers; enabled servers expose their tools to workers as callable capabilities and appear as node types on the canvas. Eulinx reuses the open MCP ecosystem to grant the "missing capabilities" (browser, web search, media, publish, research) rather than building each one, and users can write their own MCP servers. MCP tools are surfaced through the same ToolRegistry as built-in tools and are subject to the same permission gating (ADR-019); MCP trust boundaries are treated with the same sandboxing rigor as plugins (ADR-007).

## Consequences

Positive:

- Immediate breadth of capability from a growing ecosystem, without Eulinx building and maintaining each integration.
- A uniform mechanism: MCP tools become Tools in the ToolRegistry and nodes on the canvas, so they compose with everything else.
- No vendor lock-in for capabilities; users choose and self-host servers, consistent with local-first and BYOK values.

Negative:

- MCP servers are external code and a trust boundary; they must be authorized, permission-gated, and sandboxed, not blindly trusted, which is real security work.
- Reliability depends on third-party servers; an unreachable or misbehaving server must fail closed without stalling the runtime.
- The MCP spec and servers evolve independently; Eulinx must track compatibility and handle version and schema drift.

## Alternatives Considered

- **Build each capability natively in Eulinx.** Maximum control. Rejected as an unbounded maintenance treadmill that would slow the product and risk vendor lock-in, when a reusable standard already exists.
- **A proprietary Eulinx-only plugin API for capabilities.** Rejected because it would ignore the existing MCP ecosystem and force every capability author to target Eulinx specifically, shrinking available capabilities.
- **Trust MCP servers implicitly as first-party.** Rejected because external servers can be malicious or buggy; they must sit behind permissions and sandboxing like any untrusted code.
