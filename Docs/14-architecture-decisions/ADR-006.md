# ADR-006: Separate WorkflowEngine From ExecutionEngine

## Status

Accepted.

## Context

Eulinx turns a graph of nodes into real work. Two very different responsibilities hide inside "run the graph". One is *interpretation*: deciding which node is ready next, computing the ready set from dependencies, passing data along edges, handling pause/resume/cancel, and applying results back onto the graph. The other is *performance*: actually running one node's unit of work — picking an adapter, spawning or supervising a process, opening a PTY, capturing output, returning a result.

The tempting mistake is one component that both walks the graph and runs the work, often written as a recursive function that walks depth-first and calls handlers inline. That design cannot pause, cannot resume after a crash, cannot run branches in parallel cleanly, and cannot be replayed — all of which Eulinx requires. It also entangles graph logic (which must be deterministic and testable) with process management (which is messy, native, and I/O-bound).

## Decision

Eulinx splits these into two components with a hard boundary. The **WorkflowEngine** *interprets*: it owns the run object, the graph model, ready-set computation, topological and parallel dispatch, the run context, pause/resume/cancel, determinism, and crash recovery, and it drives everything as a **tick loop over persisted state**. The **ExecutionEngine** *performs*: it runs exactly one node's work when handed an execution request, and it knows nothing about graphs. The WorkflowEngine never spawns a process, opens a PTY, or writes a file; it computes ready nodes, hands each to the ExecutionEngine, and applies the returned result.

## Consequences

Positive:

- The interpreter is deterministic and testable in isolation, which is what makes deterministic replay (ADR-020) possible.
- The performer can be exercised, retried, and cancelled without any notion of graph structure, keeping messy process management out of graph logic.
- Because the engine is a tick loop over persisted state, a run pauses, resumes, and recovers after a crash — behaviors a recursive walker cannot offer.
- Parallel branches are natural: the interpreter proposes a ready set and the Scheduler admits some of it.

Negative:

- Two components and an explicit request/result contract are more moving parts than one merged runner, which a cheap model may be tempted to collapse "to keep it simple".
- Every unit of work must be expressible as a self-contained execution request with a self-contained result, forcing discipline about what a node reads and writes.
- The boundary must be actively defended: any `spawn` inside the WorkflowEngine is a bug, and reviewers must watch for the merge creeping back in.

## Alternatives Considered

- **One engine that walks and runs (recursive handler walk).** Simplest to write first. Rejected outright because it cannot pause, resume, recover, parallelize, or replay — every one of which Eulinx needs.
- **Split by node type instead of by responsibility.** Rejected because it would duplicate graph-walking logic across type-specific runners and still entangle interpretation with performance.
- **A generic job queue with no graph awareness at all.** Rejected because the ready-set/dependency semantics of a graph are exactly what must be interpreted; a flat queue loses the structure that determinism and replay rely on.
