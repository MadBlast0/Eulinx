# ADR-011: A Provider Abstraction for All AI Models

## Status

Accepted.

## Context

Eulinx must talk to many model providers — OpenAI, Anthropic, Google Gemini, OpenRouter, Ollama, LM Studio, and any OpenAI-compatible endpoint — and it is bring-your-own-key with no lock-in. These providers differ in request/response shapes, streaming formats, authentication, error semantics, rate limits, and token accounting. If worker logic, the refinement loop, and the orchestrators each spoke to providers directly, provider-specific quirks would smear across the whole codebase, adding a new provider would mean edits everywhere, and switching a run from a cheap model to a flagship would be a code change rather than a configuration change.

Eulinx's entire value proposition — using cheap models for the bulk of work and spending on quality only where it matters — depends on models being interchangeable at runtime.

## Decision

Eulinx defines a single **provider abstraction**: one internal contract for chat/completion, streaming, tool/function calling, embeddings, token accounting, retries, and fallback, behind which every provider is an adapter. Callers (workers, the refinement loop, context tooling) speak only to this abstraction, never to a provider SDK directly. Adding a provider means writing one adapter; switching models is configuration, not code. The abstraction is also where streaming, retries, fallback models, and per-call cost/token capture are implemented uniformly, so budgets (ADR-013) and profiles (ADR-012) can rely on consistent behavior.

## Consequences

Positive:

- New providers are added in one place without touching worker or orchestrator logic.
- Models are interchangeable at runtime, which is what makes the cheap-by-default, flagship-where-needed economics work and delivers on the no-lock-in promise.
- Streaming, retries, fallback, and token/cost accounting are implemented once and applied consistently, feeding budgets and profiles reliably.

Negative:

- The abstraction must be broad enough to cover real provider differences (tool calling, vision, streaming quirks) without leaking them, which is a nontrivial design and a common place to get the interface wrong.
- Provider-specific features that do not fit the common contract are either normalized (losing some fidelity) or exposed as capabilities the caller must check for, adding complexity.
- Every adapter must be kept current as providers change their APIs, an ongoing maintenance cost.

## Alternatives Considered

- **Call each provider SDK directly where needed.** Simplest per call. Rejected because provider quirks would spread everywhere, adding a provider would touch many files, and runtime model switching would become a code change — fatal to the core economics.
- **A single third-party gateway/proxy as the only interface.** Reduces our adapter work. Rejected because it can add a dependency and sometimes a hosted hop, weakening local-first/BYOK guarantees, and it constrains us to that gateway's supported providers and features.
- **Support only OpenAI-compatible endpoints.** Simpler surface. Rejected because it excludes providers with meaningfully different or richer APIs and undercuts the no-lock-in, use-any-model promise.
