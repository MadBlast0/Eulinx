# ADR-003: xterm.js Frontend With a Rust PTY Backend

## Status

Accepted.

## Context

In Eulinx, a worker is backed by a real terminal process. This is the design's core claim to being able to do *real* work — running CLIs like coding agents, executing tests, using git — rather than merely calling an API. That means Eulinx must run genuine pseudo-terminals (PTYs), stream their output live, accept input, resize them, and manage potentially many of them at once, some visible and many headless until maximized.

Two layers are required: something in the WebView that renders a terminal (colors, cursor, scrollback, selection, fit-to-container), and something native that owns the actual OS PTY, spawns the child process, and pipes bytes in both directions. The WebView cannot open a PTY itself; only native code can. This decision must also respect ADR-001's thin-Rust rule: the Rust side should be a byte pump and lifecycle manager, not a place for logic.

## Decision

Eulinx renders terminals with **xterm.js** in the React frontend and manages PTYs with a **Rust PTY layer** in the Tauri backend. Rust spawns and owns each PTY and its child process, and streams output to the frontend over Tauri's event channel; the frontend sends input and resize commands back through the service layer. A worker may keep its PTY running headless (no xterm.js instance attached) until the user maximizes it, at which point the frontend attaches an xterm.js view to the already-running stream.

## Consequences

Positive:

- Workers run in true shells, so agents can execute commands, run tests, and use existing CLI tooling — the capability that makes Eulinx's coding story real rather than simulated.
- The heavy lifting (rendering) is TypeScript via xterm.js, and the native part (PTY) is small and well-scoped, matching the thin-Rust rule.
- Headless PTYs keep many workers cheap: a hundred workers do not mean a hundred rendered terminals, only a hundred byte streams, most of them unattached.

Negative:

- PTY management, resize semantics, and cross-platform shell differences (Windows ConPTY vs Unix PTYs) are exactly the kind of native code the cheap model handles worst, so this Rust layer needs careful authorship and human review.
- Streaming high-volume output to the WebView can flood the UI; the stream must be batched/throttled, and unattached PTYs must buffer with a bounded ring so memory does not grow without limit.
- Process lifecycle (orphan cleanup on crash, on workspace close, on app exit) must be handled rigorously to avoid leaking child processes.

## Alternatives Considered

- **Fake/emulated terminals (run commands, capture stdout, no PTY).** Simpler. Rejected because interactive CLIs, TUIs, and coding agents require a real PTY; without one the product's central "real work in a real shell" claim collapses.
- **A pure-JS terminal with a Node child process (Electron-style).** Rejected as a consequence of ADR-001; Eulinx is not on Electron, and the WebView cannot spawn processes.
- **A different renderer than xterm.js.** Rejected because xterm.js is the mature, widely used, cheap-model-friendly standard for web terminals, with the addons (fit, search, links) Eulinx needs already available.
