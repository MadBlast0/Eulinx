# ADR-001: Tauri v2 with a Thin Rust Backend

## Status

Accepted. This is the foundational platform decision on which every other decision in this vault rests.

## Context

Eulinx is a desktop application that runs many AI worker terminals, a live node-graph canvas, multiple streaming model connections, and a local database, all at once, on the user's own machine. Two goals are in direct tension.

The first goal is runtime efficiency. The app must stay light on RAM and CPU because a user may run dozens of concurrent workers, each streaming tokens and possibly holding a terminal process. An Electron shell bundles a full Chromium and a Node runtime per app and typically consumes several hundred megabytes of RAM before the app does any work. For an app whose whole premise is running many things at once, that baseline overhead is a permanent tax the user pays.

The second goal is that a *cheap* coding model, DeepSeek V4 Flash, writes almost all of the code. Cheap models are strongest in TypeScript, JavaScript, HTML, and CSS, and weakest in systems languages like Rust, where they trip over the borrow checker, lifetimes, and trait bounds. A stack that forced the model to write large amounts of Rust would slow development and produce fragile code.

These pull apart: the lightest stacks (pure Rust GUI, C++) are the hardest for a cheap model; the easiest stack (Electron) is the heaviest at runtime. We need the runtime profile of a native app with a codebase that is overwhelmingly TypeScript.

## Decision

Eulinx is built on **Tauri v2**. The frontend is React 19 + TypeScript + Vite, built with pnpm. The backend is **Rust, kept deliberately thin**: it exists only to do what a WebView cannot. Rust owns PTY and process management, native filesystem access, OS secure storage, native dialogs, window management, and the small set of genuinely performance-critical operations. Everything else — orchestration logic, the workflow interpreter, memory assembly, agent reasoning glue, UI, state — lives in TypeScript. The target split is roughly 90–95% TypeScript, 5–10% Rust.

## Consequences

Positive:

- Runtime memory and startup cost are a fraction of an Electron equivalent because Tauri uses the OS WebView instead of bundling Chromium, which is exactly right for an app that runs many concurrent workers.
- The cheap coding model works almost entirely in its strongest language. Once the thin Rust bridge exists, weeks of feature work happen without touching Rust.
- Distributable binaries are small and cross-platform across Windows, macOS, and Linux from one codebase.
- The thin-backend rule creates a clean architectural seam: native concerns are physically separated from business logic.

Negative:

- Some Rust is unavoidable, and it sits on the critical path (PTYs, secure store). This code must be written carefully, likely with more human review than the TypeScript, precisely because the cheap model is weakest there.
- The OS WebView varies by platform (WebView2 on Windows, WKWebView on macOS, WebKitGTK on Linux), so rendering and behavior must be tested on each rather than against one bundled engine.
- Any temptation to "just do it in Rust because it's faster" must be actively resisted; business logic leaking into Rust erodes the very advantage this decision buys.

## Alternatives Considered

- **Electron + TypeScript (100% TypeScript).** The friendliest possible stack for a cheap model and a vast ecosystem. Rejected because its per-app Chromium+Node overhead is a permanent RAM and startup tax that is unacceptable for an app designed to run many concurrent workers.
- **Pure Rust GUI (egui / iced).** Best possible runtime performance. Rejected because a cheap coding model cannot maintain a large Rust UI codebase reliably; UI APIs are less standardized and debugging is harder for both humans and the model.
- **Flutter Desktop (Dart).** Good performance and a single language. Rejected because cheap-model support for Dart is weaker and less consistent than for React/TypeScript, and the desktop ecosystem for terminals and web-style integrations is thinner.
- **Native per-platform (SwiftUI, WinUI, Qt).** Excellent performance. Rejected because it multiplies the codebase across platforms and lands squarely in languages the cheap model handles poorly.
