# The Envelope, Field by Field

```ts
type MessageKind =
  | "task_assignment"
  | "status"
  | "question"
  | "answer"
  | "result"
  | "artifact_ready"
  | "error"
  | "heartbeat"
  | "cancel";

type MessagePriority = "low" | "normal" | "high" | "control";

type MessageEnvelope<K extends MessageKind = MessageKind> = {
  messageId: string;
  correlationId: string | null;
  causationId: string | null;
  kind: K;
  sessionId: string;
  workspaceId: string;
  fromNodeId: HierarchyNodeId;
  toNodeId: HierarchyNodeId;
  channelId: string;
  sequence: number;
  direction: "down" | "up";
  priority: MessagePriority;
  deliveryMode: "at-least-once" | "at-most-once";
  durable: boolean;
  expiresAt: string | null;
  attempt: number;
  payload: MessagePayloadFor<K>;
  sentAt: string;
};
```

```text
messageId       UUIDv7, generated by the runtime, never by the sender.
                Unique forever. Never reused. Sortable by creation time.

correlationId   The messageId of the request this message answers.
                null for messages that are not responses.

causationId     The messageId that caused this message to exist, which is not
                always the same as what it answers. A Worker's error caused by
                a task_assignment carries causationId = that assignment and
                correlationId = null. This field is for replay, not routing.

kind            The discriminant. Determines the payload type.

sessionId       Copied from the sender's node. MUST match the recipient's.

workspaceId     Copied from the sender's node. MUST match the recipient's.

fromNodeId      The authenticated sender. Set by the runtime from the
                transport identity. A sender-supplied value MUST be ignored.

toNodeId        Derived by the runtime from fromNodeId and kind. See the
                recipient derivation table below.

channelId       The parent-child channel this travels on. Derived, not given.

sequence        Monotonic per (channelId, direction), starting at 1.

direction       "down" for parent to child, "up" for child to parent.

priority        Queue ordering class. "control" jumps everything.

deliveryMode    "at-least-once" for durable kinds, "at-most-once" for
                heartbeat and status. See Part 04 for what this implies.

durable         If true, persist to SQLite before acknowledging.

expiresAt       ISO timestamp after which the message MUST be dropped
                undelivered. null means no expiry.

attempt         1 on first send, incremented on every retry.

payload         Kind-specific body. Validated at send time.

sentAt          ISO timestamp set by the runtime at accept time.
```

# Recipient Derivation

A sender never names a recipient. The runtime derives it. This table is the whole routing policy.

```text
kind             sender must be     toNodeId is
---------------  -----------------  -----------------------------
task_assignment  an orchestrator    the named child of the sender
status           any non-root node  the sender's parent
question         any non-root node  the sender's parent
answer           an orchestrator    the child that asked
result           any non-root node  the sender's parent
artifact_ready   any non-root node  the sender's parent
error            any non-root node  the sender's parent
heartbeat        any non-root node  the sender's parent
cancel           an orchestrator    the named descendant of sender
```

`task_assignment` and `cancel` take a child id from the sender because a parent has many children. The runtime MUST verify the named node is actually a child (for `task_assignment`) or a descendant (for `cancel`) of the sender. Everything else takes no recipient parameter at all.

# Payload Types

```ts
type MessagePayloadFor<K extends MessageKind> =
  K extends "task_assignment" ? TaskAssignmentPayload :
  K extends "status"          ? StatusPayload :
  K extends "question"        ? QuestionPayload :
  K extends "answer"          ? AnswerPayload :
  K extends "result"          ? ResultPayload :
  K extends "artifact_ready"  ? ArtifactReadyPayload :
  K extends "error"           ? ErrorPayload :
  K extends "heartbeat"       ? HeartbeatPayload :
  K extends "cancel"          ? CancelPayload :
  never;
```

## task_assignment

```ts
type TaskAssignmentPayload = {
  taskId: string;
  objective: string;
  acceptanceCriteria: string[];
  scope: DelegatedScope;
  contextPackageId: string;
  expectedArtifactKinds: string[];
  budget: BudgetAllocation;
  deadlineAt: string;
  optional: boolean;
};
```

`contextPackageId` is a reference, not content. The Worker fetches the package from the ContextManager. The envelope MUST NOT carry the context body; context packages routinely exceed 100 KB and the message log is not a blob store.

`optional` is declared here, at assignment time, and this is the only moment it can be set. See rule R6 in [[WorkerHierarchy-Part05]].

## status

```ts
type StatusPayload = {
  phase: "starting" | "thinking" | "tool_call" | "writing" | "verifying";
  progressPercent: number;
  currentActivity: string;
  tokensSpent: number;
  usdMicrosSpent: number;
  toolInvocationsSpent: number;
};
```

Status is `at-most-once` and non-durable. It is allowed to be lost. If a status update is load-bearing for correctness, it is not a status update, it is a result.

`progressPercent` MUST be an integer 0 to 100 and MUST NOT decrease. A model that revises its estimate downward MUST hold the previous value and adjust `currentActivity` instead. A progress bar that goes backwards is worse than no progress bar.

## question

```ts
type QuestionPayload = {
  questionId: string;
  question: string;
  blocking: boolean;
  options: string[] | null;
  defaultIfTimeout: string | null;
  timeoutMs: number;
};
```

`blocking: true` means the Worker suspends until answered. `blocking: false` means it continues on an assumption and the answer may arrive later.

`defaultIfTimeout` MUST be non-null when `blocking` is true. A blocking question with no default is a deadlock waiting for a human who may have closed the laptop. The runtime MUST reject such a message at send time with `CommError::BlockingQuestionNeedsDefault`.

A question travels one edge up. An Orchestrator that cannot answer its child's question MUST ask its own parent, as a new question with `causationId` set to the child's. It MUST NOT forward the child's envelope. Forwarding would put a grandchild's `fromNodeId` on a channel it does not belong to.

## answer

```ts
type AnswerPayload = {
  questionId: string;
  answer: string;
  answeredBy: "parent_orchestrator" | "user" | "default_on_timeout";
  confidence: "certain" | "assumed";
};
```

`answeredBy: "default_on_timeout"` with `confidence: "assumed"` is what the runtime synthesizes when a blocking question times out. The Worker MUST be able to tell a real answer from a fallback, because a wrong assumption should be recorded in the result summary.

## result

```ts
type ResultPayload = {
  taskId: string;
  outcome: "success" | "partial" | "failure" | "cancelled";
  summary: string;
  artifactIds: string[];
  budgetFinal: BudgetAllocation;
  needsDelegation: NeedsDelegationHint | null;
  needsBudget: NeedsBudgetHint | null;
  unresolvedAssumptions: string[];
};

type NeedsDelegationHint = {
  suggestedSubObjectives: string[];
  rationale: string;
};

type NeedsBudgetHint = {
  requestedAdditionalTokens: number;
  requestedAdditionalUsdMicros: number;
  rationale: string;
};
```

`needsDelegation` and `needsBudget` are hints, not commands. A Worker that needs help says so here and stops. The parent Orchestrator decides. This is how rule D7 and rule B5 from the hierarchy spec are actually carried.

## artifact_ready

```ts
type ArtifactReadyPayload = {
  artifactId: string;
  artifactKind: string;
  targetPaths: string[];
  sizeBytes: number;
  checksum: string;
  verificationState: "unverified" | "verified" | "rejected";
};
```

This message says an Artifact exists and is registered with the ArtifactManager. It does not contain the Artifact. It does not merge the Artifact. The MergeManager merges Artifacts, and it does so on the parent's explicit instruction, never as a side effect of this message arriving.

## error

```ts
type ErrorPayload = {
  errorCode: string;
  message: string;
  recoverable: boolean;
  failedTaskId: string | null;
  attemptedRetries: number;
  diagnosticContext: Record<string, string>;
};
```

`error` expects no response. It is a report, not a request. A Worker that sends `error` with `recoverable: false` MUST then send `result` with `outcome: "failure"` and stop. The error explains, the result concludes. Do not collapse them.

## heartbeat

```ts
type HeartbeatPayload = {
  nodeState: HierarchyNodeState;
  uptimeMs: number;
  lastActivityAt: string;
  queueDepth: number;
};
```

Sent every 10000 ms. Three consecutive misses, 30000 ms of silence, marks the node unresponsive and the parent MUST decide whether to cancel it. Non-durable and lossy by design.

## cancel

```ts
type CancelPayload = {
  signalId: string;
  reason: CascadeReason;
  gracePeriodMs: number;
  salvagePartialResults: boolean;
};
```

`cancel` is `control` priority. It MUST be inserted at the head of the recipient's queue, ahead of everything already waiting. Delivering a cancel behind 40 queued status messages is the same as not delivering it.

`signalId` is the cascade signal id from [[WorkerHierarchy-Part05]] and is what makes cancel idempotent. Re-delivering the same `signalId` is a no-op.

# Send-Time Validation

```text
1.  Look up the sender's node by the authenticated transport identity.
    If not found, reject with CommError::UnknownSender.
2.  If sender.state is terminal, reject with CommError::SenderTerminal.
3.  Derive toNodeId from the recipient derivation table.
    a. For task_assignment, verify the named node is a direct child.
       If not, reject with CommError::NotAChild.
    b. For cancel, verify the named node is a descendant.
       If not, reject with CommError::CascadeOutOfSubtree.
    c. For every other kind, toNodeId = sender.parentId.
       If sender.parentId is null, reject with CommError::RootCannotSend.
4.  Verify sender.sessionId == recipient.sessionId, else
    CommError::CrossSessionMessage.
5.  Verify sender.workspaceId == recipient.workspaceId, else
    CommError::CrossWorkspaceMessage.
6.  Validate payload against the schema for kind.
    On failure, reject with CommError::InvalidPayload naming the field.
7.  If kind == "question" and payload.blocking and
    payload.defaultIfTimeout == null, reject with
    CommError::BlockingQuestionNeedsDefault.
8.  Scan the payload for anything shaped like a PermissionGrant. If found,
    reject with CommError::PermissionInMessage. Invariant M12.
9.  Resolve channelId from the (parent, child) pair.
10. Assign sequence = channel.nextSequence(direction), atomically.
11. Assign messageId = UUIDv7. Set sentAt. Set attempt = 1.
12. If durable, INSERT into worker_messages and commit BEFORE enqueueing.
13. Enqueue per the backpressure policy in Part 05.
14. Emit message.sent on the EventBus.
```

Step 12's ordering is the durability contract. Enqueue-then-persist loses messages on crash. Persist-then-enqueue can duplicate them, which is why durable kinds are `at-least-once` and MUST be idempotent on the receiving side.

# Message Table

```sql
CREATE TABLE worker_messages (
  message_id      TEXT PRIMARY KEY,
  correlation_id  TEXT,
  causation_id    TEXT,
  kind            TEXT NOT NULL,
  session_id      TEXT NOT NULL,
  workspace_id    TEXT NOT NULL,
  from_node_id    TEXT NOT NULL,
  to_node_id      TEXT NOT NULL,
  channel_id      TEXT NOT NULL,
  sequence        INTEGER NOT NULL,
  direction       TEXT NOT NULL CHECK (direction IN ('down','up')),
  priority        TEXT NOT NULL,
  delivery_mode   TEXT NOT NULL,
  durable         INTEGER NOT NULL,
  expires_at      TEXT,
  attempt         INTEGER NOT NULL,
  payload_json    TEXT NOT NULL,
  sent_at         TEXT NOT NULL,
  delivered_at    TEXT,
  acked_at        TEXT
);

CREATE UNIQUE INDEX idx_msg_channel_seq
  ON worker_messages(channel_id, direction, sequence);
CREATE INDEX idx_msg_correlation ON worker_messages(correlation_id);
CREATE INDEX idx_msg_undelivered
  ON worker_messages(channel_id) WHERE delivered_at IS NULL;
```

The unique index on `(channel_id, direction, sequence)` is what enforces invariant M5 at the storage layer. Do not rely on an in-memory counter alone; it resets on restart and the index will catch the resulting collision.
